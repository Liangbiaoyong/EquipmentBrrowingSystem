## 一、开发环境准备（Windows + WSL + Docker）

**目标**：在 Windows 上搭建与目标 Linux 一致的容器化开发环境，避免「我机器上能跑」问题。

### 1.1 启用 WSL2 并安装 Ubuntu

- 确保 Windows 版本支持 WSL2（内核 5.4+，已高于目标环境的 5.1）
- 安装 Ubuntu 22.04 LTS
- 在 WSL 内更新：
  ```bash
  sudo apt update && sudo apt upgrade -y
  ```

### 1.2 在 WSL 内安装 Docker Engine

与生产环境一致，不依赖 Docker Desktop：

```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo usermod -aG docker $USER
```

重新登录后验证：`docker run hello-world`

### 1.3 VSCode 远程开发配置

- 安装 VSCode 插件：`Remote - WSL`、`Docker`、`Extension Pack for Java`、`Volar`（Vue）
- 通过 `code .` 在 WSL 中打开项目目录
- 所有开发工作在 WSL 文件系统（`/home/user/project`）进行，避免跨文件系统性能问题

### 1.4 项目结构初始化

```bash
mkdir device-borrow-system && cd device-borrow-system
git init
mkdir backend frontend sql docs tests
touch docker-compose.yml .env .gitignore
```

## 二、项目初始化与脚手架搭建（第 1 周）

**分支**：`main` → `dev/init`

### 2.1 后端 Spring Boot 脚手架

- Spring Boot 2.7.x，Java 17
- 核心依赖：Spring Web、Spring Security、MyBatis-Plus、MySQL Driver、Redis、Validation、Actuator
- 手动添加：Knife4j、Thumbnailator、MinIO Client、JWT (jjwt)、CAS Client (Apereo)
- 多环境配置：`application-dev.yml`（直连 localhost 容器）、`application-prod.yml`（环境变量注入）
- MyBatis-Plus 代码生成器生成基础 CRUD

### 2.2 前端 Vue 3 脚手架

```bash
npm create vite@latest frontend -- --template vue
cd frontend && npm install
npm install vue-router pinia axios element-plus @element-plus/icons-vue
npm install -D sass unplugin-auto-import unplugin-vue-components
```

- 配置 Vite 代理：开发时转发 `/api` 到 `localhost:8080`
- 创建基础目录：`api/`、`views/`、`components/`、`router/`、`store/`

### 2.3 Docker Compose 开发环境

基础设施全部容器化，后端和前端在开发阶段本地运行以获得热更新：

```bash
# 仅启动基础设施容器
docker compose up -d mysql redis minio

# 终端2：后端（热更新）
cd backend && mvn spring-boot:run -Dspring-boot.run.profiles=dev

# 终端3：前端（热更新，代理到 8080）
cd frontend && npm run dev
```

### 2.4 数据库初始化

- `sql/init/01-schema.sql` — 所有表 DDL
- `sql/init/02-data.sql` — 初始分类、默认管理员
- Docker Compose 启动 MySQL 时自动执行 `docker-entrypoint-initdb.d`

### 2.5 Git 分支规范

| 分支 | 用途 |
|------|------|
| `main` | 生产就绪代码，受保护，仅合并 |
| `dev` | 日常集成分支 |
| `feature/xxx` | 功能开发分支 |
| `fix/xxx` | 问题修复分支 |

提交信息约定：`feat:` / `fix:` / `docs:` / `refactor:` / `test:` / `chore:`

## 三、分阶段开发任务

### 阶段 1：用户认证与权限基础（第 2-3 周）

**分支**：`feature/auth-security`

**任务清单**：
1. Spring Security + JWT 基础框架，配置接口白名单（CAS 回调、登录接口、Knife4j 文档）
2. CAS Client 集成（先本地 Mock 模拟 CAS 返回，后续替换真实地址）
3. `SysUser` 实体和 Mapper，CAS 用户同步逻辑（首次登录自动注册）
4. 本地账号登录（管理员），密码 BCrypt 加密
5. 角色与权限定义（`@PreAuthorize`），实现 `AuthController`
   - `POST /api/v1/auth/cas/login` — CAS 回调
   - `POST /api/v1/auth/local/login` — 本地登录
   - `GET /api/v1/auth/info` — 当前用户信息
6. 前端：登录页面（CAS 跳转 + 本地登录表单）、路由守卫、Pinia user store
7. 单元测试：JWT 生成/验证、用户同步逻辑

### 阶段 2：设备管理模块（第 3-4 周）

**分支**：`feature/device-management`

**任务清单**：
1. 设备分类 `device_category`、设备 `device`、设备图片 `device_image` 实体和 Mapper
2. 设备 CRUD 接口（MyBatis-Plus Generator），含库存计算、状态转换
3. 设备图片上传：MultipartFile → Thumbnailator 压缩 → MinIO 存储 → 返回 URL
4. 设备列表查询：分页、多条件筛选（分类、状态、关键词）、模糊搜索
5. 设备详情接口（含图片数组）
6. 设备状态手动管理（标记维修/报废）
7. 前端页面：设备管理后台（管理员）、设备浏览页（用户端，卡片列表 + 筛选 + 详情弹窗）
8. 单元测试：设备库存变更逻辑、图片压缩后大小检查

### 阶段 3：借用申请与冲突检测（第 4-5 周）

**分支**：`feature/borrow-apply`

**任务清单**：
1. `borrow_record` 实体和状态机定义
2. 借用申请接口：设备列表、起止时间、审批人，校验可用性及冲突
3. 冲突检测核心：查询同一设备在目标时段已批准/进行中的记录，计算冲突数量
4. 库存预扣减或乐观锁防止超借
5. 我的借用列表（分状态），借用详情接口
6. 前端：设备详情页发起借用、时间选择器（禁用已借出时段）、审批人选择、借用列表
7. WebSocket：提交后通知审批人
8. 并发测试：JMeter 多用户同时借用同一设备

### 阶段 4：审批流程引擎（第 5-6 周）

**分支**：`feature/approval-flow`

**任务清单**：
1. 审批流定义模型（JSON 快照存于借用单），默认二级：初审 → 终审
2. `approval_log` 表操作，审批接口（通过/驳回），自动流转至下一节点
3. 权限校验：当前用户是否为该节点的审批人
4. 全部通过 → 状态「已批准」，正式扣减库存；驳回 → 释放预扣库存
5. 前端：待审批列表、审批操作对话框、审批进度展示
6. 站内信 + WebSocket 推送通知
7. 单元测试：审批流状态机所有路径（正常通过、驳回、三级审批可选）

### 阶段 5：归还管理与图片清理（第 6-7 周）

**分支**：`feature/return-management`

**任务清单**：
1. 归还接口：上传归还照片（压缩至 5MB），记录归还时间，计算逾期天数
2. 归还后库存 +1，状态 →「已归还」
3. 损坏报修：标记损坏 → 设备状态 →「维修中」
4. 图片生命周期：上传时设置 `expire_time` = 半年后
5. 定时任务 `@Scheduled(cron = "0 0 3 * * ?")`：清理过期附件，更新数据库
6. 前端：归还操作页面、逾期标签、报修表单

### 阶段 6：数据统计与看板（第 7 周）

**分支**：`feature/dashboard-stats`

**任务清单**：
1. 聚合查询：设备总览（按状态计数）、本月借用趋势（按天）、热门设备 Top10、高频用户 Top10
2. 利用率计算：实际借出总时长 / 设备总可用时长
3. Redis 缓存统计结果（过期时间 1 小时）
4. EasyExcel 导出报表
5. 前端：ECharts 图表展示、仪表盘页面

### 阶段 7：系统管理与日志审计（第 7-8 周）

**分支**：`feature/admin-audit`

**任务清单**：
1. 用户管理 CRUD、角色分配、本地账户创建
2. 审批流配置页面（系统管理员设置审批节点和人员范围）
3. AOP 切面记录操作日志（用户、操作、参数、结果、IP、耗时）
4. 日志查询界面（条件筛选）
5. 前端：系统管理菜单栏（仅管理员可见）

## 四、从 WSL 开发环境迁移到国产 Linux 部署

**目标服务器**：银河麒麟 V10（内核 5.1+），已安装 Docker 及 Docker Compose。

### 4.1 代码推送与构建

```bash
# WSL 中合并分支并推送
git checkout main
git merge dev
git push

# 目标服务器上拉取
ssh root@target-server
git clone https://github.com/Liangbiaoyong/EquipmentBrrowingSystem.git
cd EquipmentBrrowingSystem
```

### 4.2 配置调整

- 修改 `.env` 中的密码、JWT 密钥为生产强密码
- 确认 `application-prod.yml` 中 CAS 真实地址
- 配置 `nginx.conf` 的 `server_name`（如有域名）

### 4.3 镜像构建与启动

```bash
docker compose up -d --build
docker compose logs -f backend  # 确认无错误
```

- 首次启动时 MySQL 初始化脚本自动执行
- MinIO 需手动创建 `device-borrow` bucket

### 4.4 数据备份脚本

```bash
#!/bin/bash
# MySQL 数据卷备份
docker run --rm -v equipment-borrow_mysql-data:/data \
  -v /backup/mysql:/backup alpine \
  tar czf /backup/mysql_$(date +%F).tgz -C /data .

# MinIO 数据卷备份
docker run --rm -v equipment-borrow_minio-data:/data \
  -v /backup/minio:/backup alpine \
  tar czf /backup/minio_$(date +%F).tgz -C /data .
```

加入 crontab 定时执行。

### 4.5 上线检查清单

- [ ] 所有接口测试通过（Postman 集合针对生产地址运行）
- [ ] CAS 集成联调：校园网环境下单点登录跳转正常
- [ ] 文件上传与访问：MinIO 图片通过后端代理正常显示
- [ ] 用户角色权限验证
- [ ] 并发测试（模拟 30 用户）
- [ ] 定时任务确认正常运行
- [ ] 备份脚本部署并验证恢复流程

## 五、反思与优化

- **开发体验**：WSL 内 Docker + VSCode 远程开发减少环境差异，Hot Reload 保持高效
- **数据库迁移**：利用 Docker 初始化脚本保证开发/测试/生产数据库一致
- **审批流灵活性**：JSON 快照存储在借用单中，历史记录可追溯
- **图片生命周期**：定时清理 + 归还时明确设置过期时间，数据库与存储双清理
- **扩展预留**：接口前缀 `/api/v1`、模块划分清晰，便于微服务拆分
