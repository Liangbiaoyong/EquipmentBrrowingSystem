## 三、模块1：用户认证与权限基础

### 3.1 CAS 认证流程（最终实现方案）

不走标准 CAS Client 协议（CAS 2.0/3.0），采用 **Token 代理模式**：

```
浏览器 → 跳转 CAS 登录 (newcas.gzhu.edu.cn)
     → CAS 回调返回 token
     → POST /api/v1/auth/cas/login {token, cookies}
     → 后端调 https://libbooking.gzhu.edu.cn/ic-web/auth/userInfo 验证
     → 解析 JSON，提取用户身份
     → 首次登录自动建用户，重复登录更新信息
     → 签发自有 JWT 返回
```

**选择 Token 代理模式的理由**（基于 TEMP 目录的 CAS 实现分析）：

- 广州大学 CAS 通过图书馆认证中心中转，不是标准 CAS 协议
- crypto 加密方式（des.js 的 strEnc）在 Java 中缺乏直接等价实现
- Token 代理模式零依赖 CAS 客户端库，运维简单

### 3.2 学生/教师识别规则

| 判断方法                      | 学生                       | 教师               |
| ----------------------------- | -------------------------- | ------------------ |
| `ident` 字段                | `257`                    | `259`            |
| `deptName` vs `className` | 不一致（学院名 ≠ 班级名） | 一致（均为学院名） |

代码中优先以 `ident` 字段为准。

### 3.3 数据库关键设计

`sys_user` 表新增 10 个 CAS 专属字段：

```
cas_uuid      → CAS 系统全局唯一 UUID
acc_no        → CAS 账号编号
class_name    → 班级名称
class_id      → 班级 ID
dept_id       → 学院/部门 ID
ident         → 身份标识（257=学生，259=教师）
card_no       → 一卡通号
sex           → 性别
expired_date  → 账号过期日期
last_cas_login→ 最近 CAS 登录时间
```

**设计原则**：

- `auth_source='C'`（CAS）的账户 password 为空，不从密码登录
- `auth_source='L'`（Local）的账户走 BCrypt 密码验证
- 实验室管理员通过手动提升 userType 实现，不会被后续 CAS 登录降级

### 3.4 JWT 安全模型

```
Token 结构：
  Header:  { "alg": "HS512" }
  Payload: {
    "sub":     "用户ID",
    "username": "登录名",
    "roles":    ["ROLE_STUDENT", ...],
    "userType": 0,
    "iat":      签发时间,
    "exp":      过期时间（+4h）
  }
  Signature: HMAC-SHA512(header.payload, secretKey)

Principal: JwtUserPrincipal { userId, username, userType, roles, authorities }
Credentials: null（不存储原始JWT，防止日志泄露）
```

### 4.2 自动分类算法

```
输入: 国标分类名（如 "其他照相机及器材"）

Step 1: 预处理
  original    = "其他照相机及器材"
  normalized  = "照相机及器材"     // 去掉前缀"其他"

Step 2: 规则遍历（按 priority ASC）
  规则表:
    priority=50:   "照相机" → category_id=2 (摄影摄像)
    priority=100:  "计算机" → category_id=1 (计算机及外设)
    ...

Step 3: 首次包含匹配命中即返回
  规则 "照相机" 匹配 "照相机及器材" ✅
  → 返回 category_id = 2

Step 4: 全未命中
  → 返回 null → 调用方归入 category_id=10 (其他设备)
```

**设计要点**：

- 优先级机制：精确匹配词（如"无人机"priority=50）优先于泛匹配词（如"仪器"priority=100）
- 预处理去"其他"："其他照相机及器材"→"照相机及器材"，命中"照相机"关键词
- 兜底归入"其他设备"：无法分类的自动归入 category_id=10
- 管理员可增删改规则，无需重启服务

### 4.4 CSV/XLSX 格式要求

| 格式 | 编码/要求                      | 解析方式           |
| ---- | ------------------------------ | ------------------ |
| CSV  | UTF-8 编码，逗号分隔，引号转义 | 自定义状态机解析器 |
| XLSX | Excel 2007+                    | Apache POI 5.2.5   |

**关键解析能力**：

- 金额字段（含千分位逗号如 `"17,500"`）→ BigDecimal
- 日期字段：自动识别 Excel 序列号（如 `46182`）和文本日期（如 `2026-03-15`）
- 空值安全：null 字段不入库，保留数据库默认值

### 4.5 导入去重策略

```
按 asset_no 查找已有记录:
  存在 → UPDATE（更新所有字段，保留原 id）
  不存在 → INSERT（新建记录）

并发安全：
  INSERT 失败 (DuplicateKeyException) → 重新 SELECT → UPDATE
```

## 五、模块3：借用审批与权限体系

### 5.1 权限体系设计

采用 `module:action` 细粒度命名，16 个权限标识。用户类型 → 权限列表由 `PermissionConstants` 集中映射：

| 用户类型         | 权限数 | 典型权限                                                                                     |
| ---------------- | :----: | -------------------------------------------------------------------------------------------- |
| STUDENT (0)      |   8   | `device:view` `borrow:create` `borrow:my` `borrow:return`                            |
| TEACHER (1)      |   10   | 学生全部 +`approval:first` `statistics:view`                                             |
| LAB_ADMIN (2)    |   12   | `device:manage` `approval:first` `approval:second` `return:manage` `repair:manage` |
| SYSTEM_ADMIN (3) |   16   | 全部权限 +`admin:user` `admin:config` `admin:log` `admin:backup`                     |

权限在 `JwtAuthenticationFilter` 中自动注入到 `Authentication.authorities`，Controller 用 `@PreAuthorize("hasAuthority('device:manage')")` 做方法级拦截。

### 5.2 借用申请与冲突检测

```
POST /api/v1/borrows
{
  "deviceId": 1,
  "startTime": "2026-07-10T09:00:00",
  "endTime": "2026-07-10T17:00:00",
  "reason": "课程设计需要",
  "approverId": 5   // 指定审批人（教师）
}
```

冲突检测算法：查询同一设备在目标时段内是否存在以下状态的借用 →

- `PENDING_APPROVAL` / `APPROVED` / `BORROWING` / `OVERDUE`

### 5.3 审批流程引擎

默认二级审批，审批流定义以 JSON 快照存入 `borrow_record.approve_flow_def`：

```
申请人提交 → 指定审批人（教师）= 一级
           → 通过 → 实验室管理员 = 二级
                  → 通过 → APPROVED（库存-1）
           → 任一驳回 → REJECTED（终止）
```

**admin 特殊处理**：系统管理员（userType=3）拥有 `approval:first` 和 `approval:second`，可处理任一级审批。实验室管理员（userType=2）通过额外注入 `approval:first` 也可处理一级。

### 5.4 归还管理

- 归还登记：记录实际归还时间 + 设备状况描述
- 逾期自动计算：`Duration.between(endTime, now).toDays()`
- 库存恢复：`available_qty + 1`
- 损坏处理：有 `damageReport` → 设备标记「维修中」

### 5.5 设备图片管理

| 功能     | 说明                                                                |
| -------- | ------------------------------------------------------------------- |
| 添加图片 | `POST /devices/{id}/images?imageUrl=...` → 无封面时自动设为封面  |
| 删除图片 | `DELETE /devices/images/{id}` → 删除封面图自动用下一张替代       |
| 缺图查询 | `GET /devices/missing-images` → 查询无图片的设备（用于管理整改） |

### 5.6 统计与系统管理

- **仪表盘** `GET /statistics/overview`：设备总数/正常/维修/报废 + 借出/逾期/待审批数量
- **用户管理** `GET/POST/PUT /admin/users`：列表/创建/角色变更/启用禁用

## 六、模块4：设备详情增强 + 借用时长限制 + 系统配置管理

### 6.1 需求细化过程

原始需求：设备可以借用。实际细化后：

- **借用时长**：最大1周（7天），任何时间段均可（不限于工作时间）
- **时长限制必须可配置**：不能硬编码，Admin 后台可运行时修改，无需重启
- **设备详情**：不能只返回裸设备字段，必须展示图片、当前借用状态、历史借用次数

### 6.2 借用时长限制（双源配置）

```
优先级：数据库 system_config 表 > application.yml 默认值

提交借用时：
  1. configService.getIntValue("borrow.max_days", 7)  → 读取有效期上限
  2. ChronoUnit.DAYS.between(start, end)              → 计算借用天数
  3. borrowDays > maxDays → 拒绝并提示"最多 X 天"
```

配置修改方式：

- **application.yml**：`borrow.max-days: 7`（默认值，部署时设定）
- **Admin API**：`PUT /admin/config/borrow.max_days?value=14`（运行时修改，即时生效）
- **Admin API**：`DELETE /admin/config/borrow.max_days`（恢复默认值）

### 6.3 系统配置表设计

```sql
system_config (
  id           bigint PK,
  config_key   varchar(100) UNIQUE,   -- borrow.max_days
  config_value varchar(500),          -- 7
  description  varchar(200)           -- 借用最大天数
)
```

**优先级逻辑**（SystemConfigServiceImpl）：

1. `SELECT config_value FROM system_config WHERE config_key = ?`
2. 有值 → 返回数据库值
3. 无值 → 返回 `defaultValue`（来自 `@Value("${borrow.max-days:7}")`）

### 6.4 设备详情 DeviceDetailVO

```
GET /api/v1/devices/{id}  → 返回：
{
  "device":           { 设备基本信息 },
  "images":           [ 图片列表（已排序） ],
  "categoryName":     "计算机及外设",
  "isBorrowing":      true/false,
  "currentBorrower":  "用户ID:5",
  "expectedReturnTime":"2026-07-14T17:00:00",
  "borrowCount":      12
}
```

查询逻辑：

- `device_image` 表按 `sort ASC` 获取图片
- `device_category` 表获取分类中文名
- `borrow_record` 表查当前 BORROWING/OVERDUE 状态（`isBorrowing`）
- `borrow_record` 表 COUNT 获取历史借用次数

### 6.5 Admin 系统配置 API

| 方法   | 路径                                        | 说明                   | 权限             |
| ------ | ------------------------------------------- | ---------------------- | ---------------- |
| GET    | `/admin/config`                           | 所有配置列表           | `admin:config` |
| GET    | `/admin/config/{key}`                     | 单个配置值             | `admin:config` |
| PUT    | `/admin/config/{key}?value=&description=` | 修改/新增配置          | `admin:config` |
| DELETE | `/admin/config/{key}`                     | 删除配置（恢复默认值） | `admin:config` |

使用示例：

```bash
# 修改借用天数上限为14天
curl -X PUT "http://localhost:8080/api/v1/admin/config/borrow.max_days?value=14" \
  -H "Authorization: Bearer $TOKEN"

# 查看当前配置
curl "http://localhost:8080/api/v1/admin/config/borrow.max_days"
# → {"key":"borrow.max_days","value":"14"}
```

### 6.6 当前配置项

| 配置键                            | 默认值 | 说明                   |
| --------------------------------- | :----: | ---------------------- |
| `borrow.max_days`               |   7   | 借用最大天数，超出拒绝 |
| `borrow.default_approval_steps` |   2   | 默认审批级数           |

后续可扩展更多运行时配置项（如通知开关、图片保留天数等），无需改代码。

## 七、模块5：运维安全与通知体系

### 7.1 定时清理策略

每天凌晨3点执行（`@Scheduled(cron = "0 0 3 * * ?")`），分为四类操作：

| 清理项                         |    保留周期    | 配置键                        | 可运行时修改 |
| ------------------------------ | :------------: | ----------------------------- | :----------: |
| 通知(notification)             | **15天** | `cleanup.small_record_days` |      ✅      |
| 操作日志(sys_log)              | **15天** | `cleanup.small_record_days` |      ✅      |
| 审批记录(approval_log，已完成) | **15天** | `cleanup.small_record_days` |      ✅      |
| 附件/借用归还图片(attachment)  | **30天** | `cleanup.large_file_days`   |      ✅      |

**注意**：借用单(borrow_record)本身及其 approve_flow_def JSON 快照**永久保留**，不做清理。

额外执行：

- 逾期检测：BORROWING 且 endTime < now → 标记 OVERDUE + 通知
- 归还提醒：明天到期的借用 → 提前通知借用人

### 7.2 登录频率限制

Redis 滑动窗口实现：

- 每 IP 每分钟最多 **5 次**登录尝试
- 超限后锁定 **5 分钟**
- 登录成功自动清除计数

### 7.3 Token 黑名单

```
POST /auth/logout → Token 加入 Redis 黑名单（有效期=Token剩余时间）
JwtAuthenticationFilter → 每次请求先查黑名单 → 命中则拒绝
```

用途：登出后Token立即失效、管理员修改角色后废掉旧Token。

### 7.4 操作日志审计

AOP 切面自动拦截所有 `@PostMapping`/`@PutMapping`/`@DeleteMapping` 方法：

- 记录：用户ID、用户名、方法签名、参数（截断500字符）、IP、耗时、成功/失败
- 查询：`GET /admin/logs?username=&status=`（仅 SYSTEM_ADMIN）

### 7.5 当前系统配置项

| 配置键                            | 默认值 | 说明           |
| --------------------------------- | :----: | -------------- |
| `borrow.max_days`               |   7   | 借用最大天数   |
| `borrow.default_approval_steps` |   2   | 默认审批级数   |
| `cleanup.small_record_days`     |   15   | 小记录保留天数 |
| `cleanup.large_file_days`       |   30   | 大文件保留天数 |

全部可通过 `PUT /admin/config/{key}` 运行时修改，无需重启。

## 九、开发规范（从实际开发中沉淀）

### 9.1 代码规范

- 所有 API 返回统一使用 `R<T>` 封装，错误走异常而非手动构造失败响应
- Controller 只做参数校验+路由，业务逻辑全部在 Service
- 私有辅助方法需有清晰 JavaDoc，公开方法有 `@ApiOperation`
- 字段映射使用 MyBatis-Plus 的 `map-underscore-to-camel-case: true`
- 时间字段统一用 `LocalDateTime` / `LocalDate`，不使用 `Date`

### 9.2 安全规范

- 认证过滤器不存储原始 JWT 到 credentials（防日志泄露）
- Principal 使用自定义 `JwtUserPrincipal` 而非裸 Long
- `/actuator/**` 仅公开 `/actuator/health`，其余需认证
- 密钥注入用 `@Value`，长度不足 64 字节启动报错
- 登录端点 Redis 滑动窗口频率限制（5次/分钟→锁定5分钟）
- 登出 Token 立即失效（Redis 黑名单）
- 密码最低 8 位，用户名最低 3 字符

### 9.3 测试规范

- 每个 Service 至少覆盖：正常路径、异常抛出、边界条件
- 不依赖 `Thread.sleep` 测试超时（用 0ms 过期 token）
- Controller 测试不满足于 `status().isOk()`，必须验证 JSON 响应体
- Mock 使用：Service 层 `@Mock`；Controller 层 `@MockBean`
- 避免测试私有方法通过反射调用（耦合实现细节）

### 9.4 提交规范

- 每完成阶段性成果后自动 git add + commit + push
- 提交类型前缀：`feat` / `fix` / `docs` / `refactor` / `test` / `chore`
- 测试 + 代码审查通过后方可提交
- 提交前必需 `mvn test` 全通过
