## 二、阶段1 → 阶段2 的需求细化过程

### 2.1 初始设计（原始需求文档）

原始设计文档中 `device_category` 表仅有5个示例分类（测量仪器、摄影器材、模型工具、计算机设备、实验设备），`device` 表仅含核心借用字段。

### 2.2 实际数据推动细化

分析学校资产系统导出的 `20260707-在账资产查询.csv`（2470行、49列）后发现：

| 发现 | 影响的设计变更 |
|------|---------------|
| 国标分类名有 **142 种**不同值 | 无法手工维护，需要自动映射规则引擎 |
| 资产数据含大量财务管理字段（单价、金额、发票号等） | `device` 表需扩展为资产全量表 |
| 部分字段名与借用场景不完全匹配 | 设计上区分「资产管理」与「设备借用」两个视角 |
| 购置日期为 Excel 序列号格式 | 导入服务需支持日期格式自动识别 |

## 三、模块1：用户认证与权限基础

### 3.1 CAS 认证流程（最终实现方案）

不走标准 CAS Client 协议（CAS 2.0/3.0），采用 **Token 代理模式**：

```
浏览器 → 跳转 CAS 登录 (newcas.gzhu.edu.cn)
     → CAS 回调返回 token
     → POST /api/v1/auth/cas/login {token, cookies}
     → 后端调 https://libbooking.gzhu.edu.cn/ic-web/auth/userInfo 验证
     → 解析 JSON，提取用户身份
     → 首次登录自动建用户，重复登录更新信息
     → 签发自有 JWT 返回
```

**选择 Token 代理模式的理由**（基于 TEMP 目录的 CAS 实现分析）：
- 广州大学 CAS 通过图书馆认证中心中转，不是标准 CAS 协议
- crypto 加密方式（des.js 的 strEnc）在 Java 中缺乏直接等价实现
- Token 代理模式零依赖 CAS 客户端库，运维简单

### 3.2 学生/教师识别规则

| 判断方法 | 学生 | 教师 |
|----------|------|------|
| `ident` 字段 | `257` | `259` |
| `deptName` vs `className` | 不一致（学院名 ≠ 班级名） | 一致（均为学院名） |

代码中优先以 `ident` 字段为准。

### 3.3 数据库关键设计

`sys_user` 表新增 10 个 CAS 专属字段：

```
cas_uuid      → CAS 系统全局唯一 UUID
acc_no        → CAS 账号编号
class_name    → 班级名称
class_id      → 班级 ID
dept_id       → 学院/部门 ID
ident         → 身份标识（257=学生，259=教师）
card_no       → 一卡通号
sex           → 性别
expired_date  → 账号过期日期
last_cas_login→ 最近 CAS 登录时间
```

**设计原则**：
- `auth_source='C'`（CAS）的账户 password 为空，不从密码登录
- `auth_source='L'`（Local）的账户走 BCrypt 密码验证
- 实验室管理员通过手动提升 userType 实现，不会被后续 CAS 登录降级

### 3.4 JWT 安全模型

```
Token 结构：
  Header:  { "alg": "HS512" }
  Payload: {
    "sub":     "用户ID",
    "username": "登录名",
    "roles":    ["ROLE_STUDENT", ...],
    "userType": 0,
    "iat":      签发时间,
    "exp":      过期时间（+4h）
  }
  Signature: HMAC-SHA512(header.payload, secretKey)

Principal: JwtUserPrincipal { userId, username, userType, roles, authorities }
Credentials: null（不存储原始JWT，防止日志泄露）
```

### 3.5 代码审查发现的关键问题及修复

| 严重性 | 数量 | 典型问题 |
|--------|:----:|----------|
| CRITICAL | 2 | CAS API 验证 `&&`→`||` 逻辑错误；Knife4j 4.x 不兼容 SB 2.7 |
| HIGH | 8 | RestTemplate 无超时；并发 CAS 创建竞态；Principal 类型不标准 |

  id              bigint PK,

  keyword         varchar(100) NOT NULL,   -- 匹配关键词
  category_id     bigint NOT NULL,          -- 业务分类ID

  is_active       tinyint DEFAULT 1         -- 启用/禁用

### 4.2 自动分类算法

```
输入: 国标分类名（如 "其他照相机及器材"）

Step 1: 预处理
  original    = "其他照相机及器材"
  normalized  = "照相机及器材"     // 去掉前缀"其他"

Step 2: 规则遍历（按 priority ASC）
  规则表:
    priority=50:   "照相机" → category_id=2 (摄影摄像)
    priority=100:  "计算机" → category_id=1 (计算机及外设)
    ...

Step 3: 首次包含匹配命中即返回
  规则 "照相机" 匹配 "照相机及器材" ✅
  → 返回 category_id = 2

Step 4: 全未命中
  → 返回 null → 调用方归入 category_id=10 (其他设备)
```

**设计要点**：
- 优先级机制：精确匹配词（如"无人机"priority=50）优先于泛匹配词（如"仪器"priority=100）
- 预处理去"其他"："其他照相机及器材"→"照相机及器材"，命中"照相机"关键词
- 兜底归入"其他设备"：无法分类的自动归入 category_id=10
- 管理员可增删改规则，无需重启服务

### 4.3 批量导入接口

### 4.4 CSV/XLSX 格式要求

| 格式 | 编码/要求 | 解析方式 |
|------|----------|----------|
| CSV | UTF-8 编码，逗号分隔，引号转义 | 自定义状态机解析器 |
| XLSX | Excel 2007+ | Apache POI 5.2.5 |

**关键解析能力**：
- 金额字段（含千分位逗号如 `"17,500"`）→ BigDecimal
- 日期字段：自动识别 Excel 序列号（如 `46182`）和文本日期（如 `2026-03-15`）
- 空值安全：null 字段不入库，保留数据库默认值

### 4.5 导入去重策略

```
按 asset_no 查找已有记录:
  存在 → UPDATE（更新所有字段，保留原 id）
  不存在 → INSERT（新建记录）

并发安全：
  INSERT 失败 (DuplicateKeyException) → 重新 SELECT → UPDATE
```

## 六、开发规范（从实际开发中沉淀）

### 6.1 代码规范

- 所有 API 返回统一使用 `R<T>` 封装，错误走异常而非手动构造失败响应
- Controller 只做参数校验+路由，业务逻辑全部在 Service
- 私有辅助方法需有清晰 JavaDoc，公开方法有 `@ApiOperation`
- 字段映射使用 MyBatis-Plus 的 `map-underscore-to-camel-case: true`
- 时间字段统一用 `LocalDateTime` / `LocalDate`，不使用 `Date`

### 6.2 安全规范

- 认证过滤器不存储原始 JWT 到 credentials（防日志泄露）
- Principal 使用自定义 `JwtUserPrincipal` 而非裸 Long
- `/actuator/**` 仅公开 `/actuator/health`，其余需认证
- 密钥注入用 `@Value`，长度不足 64 字节启动报错
- 登录端点需加频率限制（待实施）
- 密码最低 8 位，用户名最低 3 字符

### 6.3 测试规范

- 每个 Service 至少覆盖：正常路径、异常抛出、边界条件
- 不依赖 `Thread.sleep` 测试超时（用 0ms 过期 token）
- Controller 测试不满足于 `status().isOk()`，必须验证 JSON 响应体
- Mock 使用：Service 层 `@Mock`；Controller 层 `@MockBean`
- 避免测试私有方法通过反射调用（耦合实现细节）

### 6.4 提交规范

- 每完成阶段性成果后自动 git add + commit + push
- 提交类型前缀：`feat` / `fix` / `docs` / `refactor` / `test` / `chore`
- 测试 + 代码审查通过后方可提交
- 提交前必需 `mvn test` 全通过
